AI Wizard Patterns

A reference repo and article distilling the universal UX contracts every team re-discovers when they ship a multi-step AI-driven flow. Architectural, not product-specific. Built to be stolen from, not installed.

For the next session: this doc is the brief. Open it side-by-side with your editor.

Section 1

What is an AI wizard, and why people use them

A wizard is a UX shape: multi-step, structured, guided. Stripe Connect, TurboTax, GitHub repo setup, Typeform. The user walks through a defined sequence to produce a defined output.

An AI wizard is that same shape, but the LLM lives inside the steps — not in charge of them. Three modes are worth distinguishing:

ShapeWho decides next stepWho interprets input
Plain form / wizardCodeCode (regex, dropdown)
Pure AI chatLLMLLM
AI wizard (hybrid)CodeLLM

Forms are too rigid; users type what they think the system wants, not what they mean. Chat is too unstructured; great for exploration, terrible when you need five specific fields filled in. The AI wizard is the bridge — structured output, conversational input.

Where this shape is shipping in 2026

  • Onboarding — "describe your team, we'll set up the workspace" (Linear, Notion, most B2B SaaS)
  • Planning tools — OKRs, roadmaps, product specs, business plans
  • Form-fill replacements — taxes, insurance, legal intake
  • Interview-style intake — user research, hiring, support, medical
  • Configuration — "describe your API, we'll generate the schema"
  • Adaptive education — courses that branch based on what the user says they know

It is not a fad shape. It is the natural answer to "we have an LLM and we need usable structured data out the other end." The failure modes — silent rewrites, FSM cursor desync, auto-resume confusion, missing confirmation on destructive AI actions — are universal to the shape. Every team re-discovers them the hard way. That is the gap this repo fills.

Section 2

The architecture: deterministic shell, AI core, bridge in the middle

The whole point of this architecture is that questions and sequencing are owned by code, and understanding the user's words is owned by the model. The bridge in the middle holds the UX contracts that keep the two layers in sync from the user's point of view.

Deterministic. Code owns it. Reproducible. Testable. No LLM in the loop.
Bridge. Where the UX contracts live. The most important file in the system.
AI. Interprets free text. Proposes. Never decides flow.

The mental model in one sentence: the FSM walks a tree on a fixed grammar of step ids; the AI only ever runs inside a step to interpret what the user typed; the client bridge is the only place that knows when to show the AI's proposal to the user before letting it land.

Section 3

The five UX contracts every AI wizard needs

Each one came out of a real failure mode. Generalised here so the patterns travel — the reference repo will implement each with a minimal example.

1
Pattern · persistence

Save the draft. Never auto-resume it.

Why: the FSM resumes mid-question. The user remembers the last screen they saw. These are different mental models. An auto-resumed FSM looks like a wizard that silently skipped questions.
  • Persist progress to localStorage / server.
  • Expose a Resume previous session CTA on next open.
  • Default: fresh start. Hydration is an explicit user action.
2
Pattern · confirmation

If the AI rewrote the user's input, ask before applying it

Why: silent rewrites — splitting "X and Y" into two items, polishing phrasing, normalising format — feel like gaslighting. Users stop trusting their own input.
  • AI returns a proposal, not a mutation.
  • Bridge holds the proposal as pending state; UI renders an editable confirmation card.
  • User edits / removes / collapses items, then confirms.
  • Apply as one batch — no per-item LLM round-trip.

This is the single most important contract in the system. Most products skip it and rely on an undo toast. Undo is recovery; confirmation is prevention.

3
Pattern · editability

Every AI-generated item is independently editable, re-askable, deletable

Why: AI-generated structured output becomes a frozen artifact the user can't refine without rewinding the whole flow. The contract: anything AI wrote, the user can rewrite without losing the rest of the plan.
  • Inline edit — direct text edit, no LLM turn burned.
  • Re-ask — overwrites in place, does not rewind downstream.
  • Regenerate with AI — explicit user action, not a silent retry.
  • Delete — gated by Pattern 5.
4
Pattern · cursor sync

Detect orphan questions when the user mutates state mid-flow

Why: the FSM has emitted a question about entity X. The user just deleted X from the plan strip. The question is now nonsense. Without this layer, you ship a wizard that asks questions about rows that don't exist.
  • Every question id maps to the snapshot entity it depends on (a dependency graph).
  • After any user mutation, check whether the current question's dependency still exists.
  • If missing: auto-skip via synthetic FSM reply — no user-visible "huh?" moment.

This is the layer most teams don't even know they're missing. It is the single most differentiated pattern in the set — make it the demo headline.

5
Pattern · destructive actions

Confirm destructive actions — even when the user "asked for it"

Why: in an AI flow, delete cascades. Removing a parent entity orphans its children. The confirmation moment is the right place to show the cost.
  • Modal-gate every destructive action (AlertDialog or equivalent).
  • Show what will cascade — not just "are you sure".
  • Undo toast is a secondary safety net, not the primary one.

Section 4

What you should not do

Don't let the LLM pick the next question. Even when it would be right 95% of the time.
Do let the LLM interpret the answer inside the question the FSM picked.
Don't ship "undo" as your only safety net for AI modifications.
Do show a confirmation step with editable items before the modification lands.
Don't auto-resume a saved draft just because you saved it.
Do persist and expose a "Resume" CTA the user has to click.
Don't trust that the FSM's pending question still matches the snapshot.
Do compute the dependency and skip orphan questions automatically.
Don't put business rules in your presentational components.
Do push every contract into the client bridge so the UI stays dumb.

Section 5

Market research summary (May 2026)

Findings from web research before committing to building this. Calibrate expectations to reality.

Saturated lanes — do not enter

"Agents vs workflows" (Anthropic-led, long tail of derivative posts). LangGraph-as-state-machine. HITL confirmation as a compliance keyword. Generic AI chat UX libraries.

Open lane — own it

The wizard-shaped AI UX playbook. People are writing about it (Tandem, Userpilot, HackerNoon) but nobody has codified the tactical patterns as a single artifact. Closest competitor: Mario Hayashi's "7 Prompt UX Patterns" — not wizard-specific.

Demand signal

Weak-to-moderate. Practitioners complain about agent reliability in the abstract, not "wizard rewrote my input." This is early-mover content, not viral demand-pull.

Realistic traction

Article: 500–3k impressions baseline; 10–50k if a credible voice in the AI builder community reshares. Repo: 200–1,500 stars over 6–12 months if executed well and paired with the article.

Section 6 · the decision

Build it as a reference repo. Do not build a framework.

verdict

Ship a copy-paste reference repo in the shadcn style

No npm install. No version coupling. No maintenance hell. Publish the patterns as readable code people steal into their own stack. Same audience, same recruiting upside, zero ongoing support burden. If it gets unexpected traction, then consider extracting a framework — but earn that path through demand, not assume it up front.

Why not a framework

  • LangGraph owns "deterministic FSM for LLMs" server-side. CopilotKit (~14k stars) and assistant-ui own React AI UX primitives. Vercel AI SDK owns chat. You'd enter as the 4th option with no team.
  • "AI wizard" is a real shape but a small slice. Most builders ship once for their product and never extract.
  • Framework maintenance is brutal — React/Next/LangGraph churn, provider SDKs, "does this work with Remix?" issues. It's a multi-year side quest.

Why the copy-paste repo works

  • shadcn proved this model. No dependency surface. Readable in one sitting.
  • The audience is small but high-signal: people building AI onboarding, structured-output AI forms, planning tools, internal LLM workflows.
  • Same recruiting and credibility upside as a framework, zero burden.
  • Pairs naturally with the article — repo is the article's reference implementation.

Who actually benefits, in order of audience size

  • Startup devs bolting LLMs onto onboarding / setup flows — biggest, fastest-growing.
  • Internal-tools engineers building structured AI workflows at larger companies.
  • LangGraph users with no frontend story.
  • Educators / content creators looking for a worked example to teach from.

Calibrate: that's thousands of potential users, not hundreds of thousands. Real, but niche. Treat the repo as a credibility artifact and a thinking exercise — not a growth play.

Section 7

Suggested repo structure

Single Next.js app. One worked example end-to-end. Each pattern in its own file, readable top to bottom. README is the article.

ai-wizard-patterns/ ├── README.md # the article lives here ├── ARCHITECTURE.md # diagram + the three-layer model ├── app/ │ ├── page.tsx # landing + link to the demo │ └── demo/ │ └── page.tsx # the worked example wizard ├── patterns/ # the heart of the repo — read these │ ├── 01-no-auto-resume.ts │ ├── 02-editable-confirmation.tsx │ ├── 03-per-item-controls.tsx │ ├── 04-orphan-detector.ts │ └── 05-destructive-confirm.tsx ├── wizard/ # the worked example, generalised │ ├── fsm.ts # step grammar + sequencing │ ├── ai.ts # split / polish / classify / suggest │ ├── bridge.ts # THE client bridge — all 5 contracts │ └── ui/ │ ├── QuestionCard.tsx │ ├── PlanStrip.tsx │ ├── ConfirmCard.tsx │ └── StepNav.tsx └── docs/ ├── why-ai-wizards.md ├── anti-patterns.md └── when-not-to-use.md

Demo domain — pick something universal, not OKR

Depersonalise from the source product. Candidates: plan a trip (destinations → activities → constraints), scope a side project (goals → stack → milestones), interview a candidate (role → criteria → questions). Pick one that maps cleanly to a tree (parent → children → grandchildren) so all five patterns get to demonstrate themselves.

Tech stack

  • Next.js (App Router) — defaults most readers know.
  • TypeScript — patterns are type-shaped; types are the documentation.
  • Anthropic SDK (and a one-line swap to OpenAI) — provider-agnostic patterns.
  • shadcn primitives for the UI — readers can copy components into their stack.
  • Zero database. localStorage for persistence. Keep the surface tiny.

Section 8

Article skeleton (lives in README.md)

11 sections. Total target length 1,800 – 2,400 words. Lead with the failure, not the architecture. Ship a Twitter thread first (one tweet per pattern with a screencap), then the long form.

  1. Hook — 30-second screen recording of a wizard misbehaving (orphan question, silent rewrite, or auto-resume confusion). One paragraph of anecdote.
  2. Diagnosis — these aren't model bugs; they're boundary bugs. The FSM's state and the user's state diverged and nothing was responsible for noticing.
  3. Architecture — drop the diagram. Walk the three layers. Name the pattern in your own words.
  4. Pattern 1 — never auto-resume drafts
  5. Pattern 2 — editable AI confirmation before apply (the most important one)
  6. Pattern 3 — per-item edit / re-ask / regenerate / delete
  7. Pattern 4 — orphan-question detection (the most differentiated one)
  8. Pattern 5 — confirm destructive actions, even AI-initiated ones
  9. Architecture revisited — re-read the diagram with the patterns loaded
  10. Anti-patterns — what you should not do
  11. Closing — open questions, what we'd do differently, who you want to hear from
Author beats to fill once you start writing: the moment each pattern clicked (a real failure you hit), one annotated screenshot per pattern, the one line that names the architecture in your voice.

Hard rules for the build

  • Do not import from Wiizworks. Start fresh; this is a public repo.
  • Patterns over abstractions. If two patterns share a util, copy the util — readers should be able to read each pattern in isolation.
  • Types as docs. Every public type gets a comment. Readers will skim types before reading code.
  • One worked example, not many. Resist the urge to add a second domain. The whole point is depth, not breadth.
  • No emojis. Wiizworks-blue palette. Same restraint as this doc.

Out of scope for v1

  • Multi-provider abstraction layer (use Anthropic, document the swap, move on).
  • Server-side persistence (localStorage is enough).
  • Auth.
  • A framework abstraction. Do not build one. See Section 6.
